# mahjong_scoreboard 设计文档

## 1. 项目定位

`mahjong_scoreboard`，中文名“雀桌记”，是一个面向家庭娱乐麻将桌场景的轻量级实时计分与结算 Web 应用。

项目核心目标是让同一张麻将桌上的玩家通过手机浏览器进入同一个房间，快速记录分数变化、实时查看当前分数，并在对局结束后生成清晰的结算结果。

项目不定位为麻将游戏平台，也不提供赌博、充值、提现或真实牌局撮合能力。系统只用于娱乐积分记录与结算辅助，与任何赌博、资金交易或其他违法行为无关。使用者应遵守所在地法律法规，不得将本项目用于任何违法违规用途。

项目 logo 使用绿色圆角图标，主体由麻将牌和积分面板组成，对应资源为：

- 前端运行资源：`web/public/logo.png`；
- 文档展示资源：`docs/assets/brand/logo.png`。

## 2. 使用场景

典型使用流程：

1. 用户在本地服务器或 Docker 环境中启动应用。
2. 玩家通过局域网访问 `http://服务器局域网IP:8899`。
3. 房主创建三人或四人房间，并设置起始分、计分单位、是否允许负分。
4. 其他玩家通过房间号、二维码链接或页面扫码加入房间。
5. 非房主玩家在等待阶段准备。
6. 房间人数已满且所有非房主玩家准备完毕后，房主收到提示并开始对局。
7. 对局过程中记录“扣分玩家给加分玩家多少分”。
8. 所有玩家页面通过 Socket.IO 实时同步当前分数和流水。
9. 房主结束对局，系统生成结算结果。
10. 用户可以在历史记录中查看已结束对局。

## 3. 功能范围

当前项目支持：

- 创建三人或四人房间；
- 从首页继续进入未完成对局；
- 超时对局可删除本机继续入口；
- 玩家通过房间号、二维码链接或页面扫码加入；
- 设置玩家昵称；
- 设置起始分；
- 设置计分单位；
- 设置是否允许负分；
- 非房主准备、取消准备、退出等待房间；
- 房主在等待阶段解散房间；
- 房间人数已满且所有非房主准备后提示房主开始对局；
- 记录扣分玩家、加分玩家、分数和备注；
- 通过弹窗记录积分变动；
- 实时同步房间状态；
- 查看记分流水；
- 相关双方确认后撤销指定流水；
- 房主结束对局；
- 自动生成结算结果；
- 查看历史对局；
- 查看历史对局详情。

当前项目不包含：

- 完整麻将规则；
- 自动识别胡牌；
- 番型计算；
- 用户注册登录；
- 微信登录；
- 公网分享；
- 充值提现；
- 复杂排行榜；
- 语音聊天；

## 4. 技术架构

项目采用前后端分离、Docker Compose 统一部署的结构。

```text
手机浏览器
  ↓
web 容器：Nginx + Vue 静态资源
  ↓ /api 与 /socket.io 反向代理
server 容器：Fastify + Socket.IO
  ↓
SQLite 数据库文件
```

### 4.1 前端

前端技术栈：

- Vue 3；
- Vite；
- TypeScript；
- Pinia；
- Vue Router；
- Socket.IO Client；
- 自定义移动端优先 CSS。

前端职责：

- 创建房间；
- 加入房间；
- 继续对局；
- 展示等待状态；
- 展示玩家当前分数；
- 提交记分操作；
- 展示流水；
- 展示最终结算；
- 展示历史记录；
- 接收 `room:sync` 实时状态。

前端不直接计算最终可信分数。所有分数变化必须由后端处理后返回。

### 4.2 后端

后端技术栈：

- Node.js 24；
- Fastify；
- Socket.IO；
- Node.js 内置 `node:sqlite`；
- SQLite。

后端职责：

- 提供 REST API；
- 提供 Socket.IO 实时同步；
- 创建和维护房间状态；
- 管理玩家；
- 执行记分事务；
- 执行撤销事务；
- 生成结算；
- 读取历史记录；
- 作为分数计算的唯一可信来源。

### 4.3 数据库

数据库使用 SQLite，默认文件路径：

```text
server/data/mahjong.db
```

Docker Compose 中该目录挂载到宿主机，保证容器重建后数据仍然保留。

当前数据表：

- `rooms`：房间；
- `players`：玩家；
- `transactions`：记分流水；
- `settlements`：结算结果。

## 6. 页面设计

### 6.1 首页

首页提供：

- 项目 logo；
- 创建房间入口；
- 加入房间入口；
- 继续对局；
- 进入历史记录。

创建页提供：

- 房间模式：三人 / 四人；
- 房间名称；
- 房主昵称；
- 起始分；
- 计分单位；
- 是否允许负分。

加入页提供：

- 房间号；
- 扫码填写房间号；
- 玩家昵称。

继续对局：

- 展示当前浏览器加入过的未完成对局；
- 创建或开始超过 12 小时的对局标记为已超时；
- 已超时对局提供移除按钮；
- 移除只清理当前浏览器本地保存的入口，不删除服务器数据。

加入房间字段：

- 房间号；
- 玩家昵称。

### 6.2 房间页

房间页根据房间状态展示不同操作。

等待状态：

- 房间号；
- 房间二维码入口；
- 当前玩家列表；
- 当前已加入人数 / 房间人数；
- 房主标识；
- 非房主准备状态；
- 普通玩家准备、取消准备、退出房间按钮；
- 房主解散房间按钮；
- 开始对局按钮。

进行中状态：

- 玩家当前分数；
- 相对起始分的输赢；
- 积分变动操作按钮；
- 积分变动弹窗；
- 房主的自己扣分 / 为他人扣分页签；
- 按钮式扣分玩家 / 加分玩家选择；
- 页面内数字键盘输入分数，默认留空并在确认时校验；
- 备注输入；
- 确认积分变动；
- 指定流水撤销确认；
- 结束结算；
- 最近流水。

已结束状态：

- 最终结算；
- 玩家最终输赢；
- 完整流水。

### 6.3 历史页

历史页展示已结束对局：

- 对局名称或房间号；
- 玩家列表；
- 结束时间。

历史详情展示：

- 结算结果；
- 完整流水；
- 已撤销流水标记。

## 7. 核心业务规则

### 7.1 房间

房间状态：

```text
waiting
playing
finished
```

房间号为 6 位大写字符，由后端生成。

等待页提供房间二维码入口。二维码内容为加入房间链接：

```text
{应用访问地址}/room/join?code={房间号}
```

外部相机或浏览器扫码打开该链接后，前端会进入加入房间页并自动填入房间号。加入房间页也提供“扫码填写”按钮，用浏览器摄像头解析二维码并填入房间号。

浏览器内扫码依赖 `getUserMedia`，通常需要 HTTPS 或 `localhost` 安全上下文。局域网 HTTP 访问时，如果浏览器禁止摄像头权限，用户仍可以使用系统相机扫码打开链接。

### 7.2 玩家

玩家加入房间时需要提供昵称。同一房间内昵称不能重复。

玩家在等待阶段退出房间后，会从房间玩家列表中删除，该昵称可以被其他设备重新使用。

非房主玩家在等待阶段可以准备或取消准备。房主不需要准备。房间人数已满且所有非房主玩家准备完毕后，房主会收到提示。

开始对局必须同时满足：

- 房间状态为等待中；
- 当前玩家是房主；
- 当前玩家数达到房间人数；
- 所有非房主玩家都已准备。

房主可以在等待阶段解散房间。解散后，后端删除房间，并通过 Socket.IO 通知房间内所有客户端返回首页。

项目没有账号系统，当前玩家身份保存在浏览器本地存储中。

### 7.3 记分

记分操作包含：

- 扣分玩家；
- 加分玩家；
- 分数；
- 备注；
- 操作人。

前端交互规则：

- 非房主只能打开自己的扣分弹窗；
- 房主可以使用“自己扣分”或“为他人扣分”两种模式；
- 扣分玩家和加分玩家使用按钮选择，不使用下拉框；
- 扣分玩家与加分玩家互斥，已选扣分玩家在加分玩家栏中不可点击；
- 分数输入默认留空；
- 分数输入使用页面内数字键盘，只提供数字、退格和收回；
- 每次确认或取消积分变动后，分数输入恢复为空；
- 不提供快捷加减按钮，避免误操作。

后端在同一个 SQLite 事务中完成：

1. 校验房间状态；
2. 校验玩家归属；
3. 校验分数；
4. 校验非房主只能以自己作为扣分玩家；
5. 扣减扣分玩家分数；
6. 增加加分玩家分数；
7. 写入流水；
8. 返回完整房间状态。

### 7.4 撤销

撤销作用于用户在流水列表中选择的指定有效流水。

撤销不会删除流水，而是：

- 记录撤销申请人；
- 记录扣分玩家确认状态；
- 记录加分玩家确认状态；
- 标记流水为已撤销；
- 记录撤销时间；
- 回滚扣分玩家和加分玩家分数。

只有该笔流水的扣分玩家和加分玩家可以确认或取消撤销。双方都确认后，后端才会在同一个事务中回滚分数并标记流水为已撤销。如果一方取消撤销，本次撤销申请失效。

### 7.5 结算

结束对局时，后端为每位玩家生成结算：

```text
输赢分 = 当前分 - 起始分
金额换算结果 = 输赢分 / 计分单位
```

所有玩家输赢分之和应为 0。

只有房主可以结束对局。

## 8. 实时同步

客户端进入房间页后发送：

```text
room:join
```

后端将客户端加入：

```text
room:{房间号}
```

每次房间状态变化后，后端广播：

```text
room:sync
```

广播内容为完整房间状态：

- room；
- players；
- transactions；
- settlements。

前端收到 `room:sync` 后直接刷新页面状态。

## 9. 部署设计

Docker Compose 包含两个服务：

- `web`：构建 Vue 静态资源，用 Nginx 对外提供访问；
- `server`：运行 Fastify 和 Socket.IO 服务。

默认端口：

```text
web: 8899
server: 3100
```

外部用户只需要访问：

```text
http://服务器局域网IP:8899
```

`web` 容器负责把 `/api/` 和 `/socket.io/` 转发到 `server` 容器。

## 10. 设计原则

- 手机端优先；
- 大按钮、大数字、少输入；
- 后端作为唯一可信状态源；
- 所有关键操作写入数据库；
- 撤销保留审计痕迹；
- 数据文件易于备份；
- 局域网部署优先；
- 功能保持轻量，避免引入复杂账号和规则系统。
