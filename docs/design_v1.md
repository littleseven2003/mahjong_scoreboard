# mahjong_scoreboard 设计文档 v1

## 1. 项目名称建议

建议项目名使用：

```text
mahjong_scoreboard
```

该名称整体是合适的，含义清晰，能够准确表达“麻将记分板 / 麻将积分结算工具”的定位。

相比其他命名：

- `mahjong-score`：更短，但含义略泛；
- `mahjong_scoreboard`：更明确，适合 GitHub 仓库、Docker 项目和 Codex 开发上下文；
- `mahjong-ledger`：偏账本，不够直观；
- `mahjong-room-score`：描述准确但略长。

因此第一版项目建议统一命名为：

```text
mahjong_scoreboard
```

中文定位建议为：

```text
麻将桌记分器 / 麻将积分结算工具 / 家庭娱乐积分记录工具
```

不建议将项目定位为“麻将游戏平台”或“赌博结算平台”，第一版只做娱乐积分记录与结算。

---

## 2. 项目目标

本项目计划开发一个可以部署在绿联云 NAS Docker 环境中的轻量级麻将记分 Web 应用。

主要使用场景为：

- 用户在局域网内通过手机访问 NAS 上部署的网页；
- 创建一个三人或四人麻将房间；
- 每位玩家通过手机进入同一个房间；
- 房主开始对局并设置起始分；
- 对局过程中可以随时记录“谁给谁多少分”；
- 页面实时同步所有玩家的当前分数；
- 对局结束后生成最终结算结果。

第一版项目重点不是自动识别麻将规则，也不是完整麻将游戏，而是提供一个简单、稳定、易用的多人实时记分工具。

---

## 3. 部署环境分析

### 3.1 目标部署环境

目标部署环境为：

```text
绿联云 NAS
Docker / Docker Compose
局域网访问优先
移动端浏览器使用
```

应用主要在麻将桌旁使用，因此优先考虑：

- 内网访问速度；
- 手机端操作体验；
- Docker 部署稳定性；
- 数据持久化；
- 多人实时同步；
- 简单维护和备份。

### 3.2 访问方式

第一阶段建议采用局域网访问：

```text
http://NAS局域网IP:端口
```

示例：

```text
http://192.168.31.100:8899
```

后续如有需要，再考虑：

- 绿联 NAS 反向代理；
- IPv6 公网访问；
- Tailscale / ZeroTier；
- 内网穿透；
- HTTPS 域名访问。

第一版不建议把公网访问作为核心目标，避免增加部署复杂度。

## 4. 项目部署方式

用户明确选择“项目部署”而不是“单容器部署”。

推荐项目结构如下：

```text
mahjong_scoreboard/
├── docker-compose.yml
├── server/
│   ├── Dockerfile
│   ├── src/
│   └── data/
│       └── mahjong.db
└── web/
    ├── Dockerfile
    └── src/
```

### 4.1 结构说明

#### 根目录

```text
mahjong_scoreboard/
```

项目根目录，包含整体 Docker Compose 配置和前后端工程。

#### docker-compose.yml

负责统一编排前端和后端服务。

建议包含：

- web 前端服务；
- server 后端服务；
- SQLite 数据挂载目录；
- 服务间网络；
- 端口映射；
- 自动重启策略。

#### server/

后端服务目录。

主要职责：

- 提供 REST API；
- 提供 WebSocket / Socket.IO 实时通信；
- 处理房间创建、加入、开始、结束；
- 处理分数变动；
- 处理撤销；
- 写入和读取 SQLite 数据；
- 作为分数计算的唯一可信来源。

#### web/

前端服务目录。

主要职责：

- 提供手机端页面；
- 创建房间；
- 加入房间；
- 显示玩家分数；
- 提交记分操作；
- 显示流水记录；
- 显示最终结算结果；
- 通过 WebSocket 接收实时更新。

## 5. 推荐技术栈

### 5.1 前端技术栈

推荐：

```text
Vue 3
Vite
TypeScript
Pinia
Vue Router
移动端优先 UI
```

选择原因：

- 适合快速开发轻量 Web 应用；
- 移动端页面实现简单；
- 状态管理清晰；
- 适合后续 PWA 化；
- 与用户已有 Vue 使用经验较契合。

前端第一版不需要复杂 UI 框架，也不需要过度设计。重点是：

- 大按钮；
- 大数字；
- 操作路径短；
- 手机端可读性好；
- 麻将桌场景下容易点按。

可选 UI 方案：

```text
Vant
Naive UI
Element Plus Mobile 适配
自定义 CSS
```

第一版更推荐 Vant 或自定义轻量样式。

---

### 5.2 后端技术栈

推荐：

```text
Node.js
Fastify 或 Express
Socket.IO
SQLite
```

更推荐：

```text
Node.js + Fastify + Socket.IO + SQLite
```

选择原因：

- Node.js 与 WebSocket / Socket.IO 生态成熟；
- 适合房间实时同步类应用；
- Fastify 性能较好且结构清晰；
- SQLite 足够支撑本项目数据量；
- 部署在 NAS 上资源占用低。

后端应承担所有核心计算逻辑，前端不应自行决定最终分数。

---

### 5.3 数据库

第一版推荐使用：

```text
SQLite
```

原因：

- 无需额外数据库容器；
- 一个文件即可持久化；
- 适合 NAS 文件备份；
- 数据量小，性能足够；
- 降低部署复杂度。

数据库文件建议放在：

```text
server/data/mahjong.db
```

并通过 Docker volume 挂载到 NAS 本地目录，避免容器重建后数据丢失。

## 6. 推荐 Docker 部署方案

### 6.1 服务拆分

第一版项目部署建议拆为两个服务：

```text
web：前端服务

```

逻辑上为前后端分离，部署上通过 Docker Compose 统一管理。

### 6.2 推荐访问链路

推荐访问链路为：

```text

    ↓

    ↓

    ↓

```

### 6.3 端口规划

建议：

```text
web 对外端口：8899

```

外部用户只需要访问 web 服务。

前端与后端通信可以通过：

```text
/api
/socket.io
```

后续可由前端容器中的 Nginx 反向代理到 server 服务。

### 6.4 数据持久化

必须持久化的内容：

```text
SQLite 数据库文件
历史对局数据
玩家与房间记录
结算记录
```

建议 NAS 挂载路径示例：

```text
/volume1/docker/mahjong_scoreboard/data
```

或根据绿联 NAS 实际路径调整。

## 7.6 流水记录

每次记分都生成一条流水。

流水字段包括：

```text
序号
付款方
收款方
分数
备注
操作人
操作时间
是否已撤销
```

示例：

```text
第 3 笔：张三 给 李四 20 分，备注：点炮
```

流水用于：

- 对账；
- 撤销；
- 回看；
- 生成最终结算。

---

## 7.7 撤销功能

第一版建议只实现：

```text
撤销上一笔
```

撤销逻辑：

- 不建议直接删除原始流水；
- 应标记原流水为已撤销；
- 分数回滚；
- 广播最新房间状态；
- 可记录撤销时间和撤销人。

这样可以避免对账混乱。

## 7.8 结束结算

点击“结束对局”后生成结算页。

结算内容包括：

```text

起始分
最终分
输赢分
金额换算结果
```

示例：

```text
张三：930，输 70
李四：1120，赢 120
王五：980，输 20
赵六：970，输 30
```

应包含总分校验：

```text
所有玩家输赢分之和应为 0
```

如果设置金额换算，则显示：

```text
1 分 = 1 元
```

或：

```text
10 分 = 1 元
```

## 10. 实时同步设计

项目需要多人手机实时同步，推荐使用 Socket.IO。

核心事件可设计为：

```text
room:create
room:join
room:sync
game:start
score:transfer
score:undo
game:finish
```

同步原则：

```text
后端是唯一可信状态源
前端只提交操作，不自行决定最终结果
每次分数变化后，后端广播完整房间状态
客户端收到 room:sync 后刷新页面状态
```

这样可以避免多个手机同时操作时出现分数不一致。


这样更适合真实麻将桌使用。

推荐项目名：

```text
mahjong_scoreboard
```
